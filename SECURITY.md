# Security policy

## Purpose

Private reporting path for vulnerabilities in this public example.

## Who should use this

Anyone who finds a vulnerability in the application, CDK stack, dependencies,
workflows, or documented security process.

## What this does not do

Does not authorize testing accounts or resources you do not own, and provides
no operational support SLA.

## Supported scope

The current `main` branch is supported. Historical commits and independently
modified forks are not maintained releases.

## Vulnerability reporting

Do not open a public issue containing an exploit, credential, account ID, SSO
URL, live resource name, object data, or private log output.

**Current reporting path:** GitHub private vulnerability reporting is not
currently enabled. Contact the repository owner through their
[GitHub profile](https://github.com/Aron-Chu) to establish a private channel
before sharing sensitive details.

**Future hardening:** Enable
[private vulnerability reporting](https://docs.github.com/en/code-security/security-advisories/working-with-repository-security-advisories/configuring-private-vulnerability-reporting-for-a-repository)
and use the repository's private security-advisory form.

Include only what is needed to reproduce and assess the issue:

- affected file, workflow, or documented control;
- impact and prerequisites;
- minimal reproduction using placeholders;
- suggested mitigation, if known; and
- whether the issue may already have exposed data or credentials.

The maintainer will assess severity and coordinate disclosure when available.
No response or remediation deadline is promised for this example repository.
